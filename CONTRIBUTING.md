# Contributing

Thank you for your interest in contributing to this project.

This repository is maintained as a proprietary internal codebase, so contributions are limited to people who have explicit permission from the project owner or maintainer.

## Before you contribute

Please make sure you:

- Have been invited or approved to contribute
- Have read and agreed to the project’s Code of Conduct
- Understand that all contributions are subject to review and approval
- Do not share repository content outside the approved team

## What kinds of contributions are welcome

Depending on your role and access, contributions may include:

- Bug fixes
- Feature work
- Documentation improvements
- Test coverage
- Refactoring and maintainability improvements
- Security and performance improvements

## Contribution process

1. Create a branch for your work.
2. Make small, focused changes where possible.
3. Add or update tests when behavior changes.
4. Ensure the code builds and relevant checks pass.
5. Open a pull request with a clear summary of:
    - what changed
    - why it changed
    - how it was tested
    - any follow-up work or trade-offs

## Coding guidelines

Please follow the conventions already used in the repository:

- Keep code readable and straightforward
- Prefer small, composable functions
- Add comments only when they clarify non-obvious logic
- Keep changes scoped to the relevant feature or module
- Preserve existing architecture and boundaries
- Update documentation when behavior changes

## Testing expectations

When you change behavior, please add or update tests as appropriate.

At a minimum, make sure your change:

- builds successfully
- passes existing tests
- does not break related flows
- includes verification steps in the pull request description

## Review process

All contributions are reviewed by the project maintainers.

A contribution may be:

- approved as-is
- approved with requested changes
- deferred for later
- rejected if it is out of scope or inconsistent with project goals

Please be prepared to revise your work based on review feedback.

## Security issues

Do not open public issues for sensitive security concerns.

If you discover a security issue, report it privately to the project maintainer or designated contact channel.
